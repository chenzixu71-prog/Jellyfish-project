# 水母diy学习助手 - 需求分析

## 0. 当前判断

这是一个面向“自定义知识输入 -> AI 生成互动题闯关 -> 即时讲解 -> 复盘报告”的 AI 学习产品需求。当前最适合先做 MVP，不宜一开始扩成完整知识库、多人 PK、VIP 商业化和多模态全能力平台。

当前项目上下文只覆盖项目 A 的早期工程骨架：Python 后端、微信小程序前端、关卡/题目/结果/个人中心页面占位。它不能代表完整商业判断，但足够支撑第一版需求沉淀。

## 1. 产品命名

- 产品名：水母diy学习助手
- 项目目录名暂不改动：`02-Python全栈AI刷题学习小程序`
- 命名含义：用户可以把自己想学的知识“DIY”成一套可闯关、可复盘、可分享的互动学习内容。

## 2. 用户

### 主要用户

- 想快速掌握一个新知识点的个人学习者。
- 非科班学习编程、AI、考试内容、语言词汇、职场知识的人。
- 有一段材料、一个网页、一个视频或一个文档，但不知道如何转成练习题的人。

### 次要用户

- 想给别人生成练习题的老师、培训者、社群运营者。
- 想把复习材料转成互动题库的考试备考用户。

### 当前不优先服务

- 企业级培训管理员。
- 大班级教学管理者。
- 追求完整 LMS、复杂权限和批量管理的机构客户。

## 3. 场景

用户看到一个想学的内容，例如一句话、一个概念、一段文章、一个网页或一个视频。他不想只读一遍，而是希望系统帮他变成可以互动答题的学习过程。

核心场景：

1. 用户输入“我想学什么”。
2. AI 根据输入内容或外部信息源整理知识点。
3. AI 生成闯关题、选项、正确答案和讲解。
4. 用户逐题作答。
5. 系统即时反馈对错和知识讲解。
6. 通关后生成复盘报告，可分享给好友。
7. 后台保留学习记录，方便后续复盘。

## 4. 需求解读

用户真正需要的不是“AI 出题”本身，而是把零散知识转成一个有反馈、有节奏、有成就感的学习闭环。

因此需求可以拆成 4 层：

| 层级 | 需求含义 | MVP 做法 |
| --- | --- | --- |
| 输入层 | 用户自由输入想学内容 | 先支持文本输入；文档/视频/网页解析后置 |
| 生成层 | AI 提炼知识并生成题目 | 先生成单选题、答案、讲解 |
| 互动层 | 用户闯关答题并获得即时反馈 | 先做关卡列表、答题页、结果页 |
| 复盘层 | 系统生成知识总结和报告 | 先做简单复盘文本；分享海报后置 |

## 5. MVP 核心功能

### 5.1 自由输入学习主题

用户可以输入：

- 一句话：例如“我想学 Git 的 pull、commit、push”。
- 一段文字：例如一段课堂笔记。
- 一个网页 URL：MVP 可先只保存 URL，不立即做复杂解析。

MVP 边界：

- 先不支持视频解析。
- 先不支持大文档上传。
- 先不做复杂 RAG 知识库。

### 5.2 AI 生成闯关题

AI 输出内容包括：

- 关卡标题。
- 题目。
- 选项。
- 正确答案。
- 答案讲解。
- 知识点总结。

MVP 题型：

- 单选题优先。
- 判断题可作为第二优先级。
- 主观题、语音题、图片题后置。

### 5.3 用户答题闯关

页面链路：

```text
首页/输入页 -> 生成中 -> 关卡页 -> 答题页 -> 结果页 -> 复盘报告
```

MVP 先保留当前页面：

- `pages/levels`：关卡页。
- `pages/quiz`：答题页。
- `pages/result`：结果页。
- `pages/profile`：个人中心页。

后续新增：

- `pages/create`：输入学习主题。
- `pages/report`：复盘报告。

### 5.4 即时反馈和讲解

每道题答完后，用户应立刻看到：

- 是否答对。
- 正确答案。
- 为什么这个选项正确。
- 相关知识点补充。

这是产品的关键体验，不能只显示分数。

### 5.5 复盘报告

通关后生成：

- 本次学习主题。
- 掌握知识点。
- 错题列表。
- 易错原因。
- 下一步复习建议。

MVP 可以先用普通文本报告，不做精美海报。

### 5.6 后台分析

用户可以看到：

- 学过哪些主题。
- 每次得分。
- 错题情况。
- 哪些知识点反复出错。

MVP 先在个人中心展示简单学习记录，复杂图表后置。

## 6. 游戏化学习设计

游戏化不是加装饰，而是利用反馈、竞争、进度和成就来提高持续学习动力。

| 游戏化元素 | 心理驱动力 | MVP 是否做 | 本项目落地方式 |
| --- | --- | --- | --- |
| 连续学习 | 损失规避、成就感 | 后置 | 后续做 streak |
| 排行榜 | 社交竞争、地位认同 | 后置 | 后续做好友榜，不做全网榜 |
| 勋章 | 自我价值确认、收集欲 | 可后置 | 完成首个主题、连续 3 天学习时发放 |
| 即时反馈 | 认知闭环、减少挫败感 | 必做 | 每题立刻讲解 |
| 关卡进度 | 目标感、完成欲 | 必做 | 题目按关卡展示 |
| 分享报告 | 社交展示、复盘动机 | MVP 可简化 | 先支持文本分享，海报后置 |

## 7. 扩展功能

以下功能先记录，不进入第一版 MVP：

1. RAG 知识库：上传资料后生成长期题库。
2. AI 生图：根据知识类型生成题目图片或选项图。
3. VIP 盈利：充值、会员、题量限制、模型额度。
4. 多人 PK：好友对战、排行榜、数字人、语音读题。
5. 多模态解析：文档、网页、视频、音频解析。

### 7.1 用户系统扩展需求

核心学习闭环已经完成后，用户系统进入下一阶段设计。用户系统的目标不是单纯增加登录入口，而是让学习数据可以长期保存、跨设备访问，并支持后续成长体系、错题复习、报告历史、额度和会员能力。

第一版用户系统范围：

- 微信小程序一键登录，使用微信 `code` 换取用户身份。
- 保留游客模式，未登录用户仍可完成核心学习链路。
- 用户登录后，将当前游客 `sessionId` 下的学习记录、报告、错题和成长数据绑定到用户账号。
- “我的”页面展示用户基础资料、学习统计、成长值、连续学习天数、报告历史和错题入口。
- 用户资料第一版只做基础展示，不做手机号、密码注册、复杂资料编辑和会员支付。

数据存储方向：

- 由于后续会保存用户、题库、答题记录、错题、报告、成长值、文件解析记录等关系型数据，不建议继续使用 SQLite 作为正式扩展方案。
- 用户系统阶段开始采用 MySQL 作为主数据库。
- Redis、对象存储、支付系统暂不进入第一版用户系统。

- 第一版要围绕 AI 刷题/闯关学习。来源：项目目录与当前工程骨架。
- 用户输入可以是一句话、一段话、文档、视频或网页。来源：PM 需求截图。
- 用户需要即时答案和知识讲解。来源：PM 需求截图。
- 复盘报告和分享是核心闭环的一部分。来源：PM 需求截图。
- 游戏化学习是重要方向，但需要分阶段落地。来源：PM 需求截图与竞品分析。
- 当前已进入用户系统扩展阶段，第一版采用微信一键登录 + 游客数据绑定。来源：PM 确认。
- 后续关系型数据正式存储采用 MySQL，不采用 SQLite 作为用户系统主数据库。来源：PM 确认。
- 第一版用户资料只做基础版，不做手机号、密码注册和复杂资料编辑。来源：PM 确认。

## 9. 待确认信息

1. MVP 是否只支持“文本输入”，还是必须支持 URL 解析。
2. 分享报告是小程序转发文本，还是要生成图片海报。
3. MySQL 部署环境使用本地 Docker、云数据库，还是云服务器自建 MySQL。
4. 微信登录使用的 AppID/AppSecret 是否由开发者本地 `.env` 配置，还是后续统一放到部署环境变量中。

## 10. 冲突点

| 冲突 | 说明 | 建议 |
| --- | --- | --- |
| 输入范围很大，但 Day 02 工程能力还很早 | 文档/视频/网页解析会显著扩大复杂度 | MVP 先做文本输入，URL 只记录不解析 |
| 游戏化方向很多，但核心学习闭环未完成 | 排行榜、PK、VIP 会分散开发注意力 | 先做即时反馈和关卡进度 |
| “AI 自动获取全网知识”存在质量与版权风险 | 搜索结果可能不准，也可能涉及内容授权 | 第一版先让用户提供文本，外部搜索后置 |

## 11. 风险提示

- 内容准确性风险：AI 生成题目和讲解可能出错，需要提供“答案可能有误/反馈错误”的机制。
- 版权风险：解析文档、视频、网页时，必须考虑用户是否有权使用这些内容。
- 成本风险：频繁生成题目和讲解会消耗模型调用额度，后续 VIP 或额度设计要和成本挂钩。

## 12. 转 PRD 前自检

结果：可以进入 PRD 草稿，但只能作为“草稿态”。

原因：

- 用户、场景、需求含义已经基本清楚。
- MVP 范围可以先收敛到文本输入、AI 出题、答题反馈、复盘报告。
- 仍有 3 个待确认点，但不阻塞第一版需求文档。
