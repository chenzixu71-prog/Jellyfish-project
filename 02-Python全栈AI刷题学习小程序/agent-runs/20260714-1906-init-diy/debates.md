## 紧急 Demo 评审：2026-07-14

### 第一轮：独立提案

- 产品：有条件通过。核心链路应保持“输入/上传 -> 知识库 -> 闯关 -> 讲解 -> 报告”，首页只保留一个主 CTA；联网状态必须显性反馈。
- 设计：有条件通过。采用克制的游戏化方案：水母承担品牌和情绪反馈，内容区保持白底高可读；拒绝所有卡片持续浮动，也拒绝装饰元素遮挡操作区。
- 前端：有条件通过。优先 CSS 位移、缩放和底部抽屉，避免第一版引入 Lottie；GPT 切图可用，但 4 张图片均超过 244 KiB，需进入性能清单。
- 测试：第一轮否决。H5 点击生成时出现 Taro Button loading 的 DOM 插入异常；未修复前不能判定 Demo 可体验。

### 第二轮：挑战与修订

- 产品挑战设计：过度克制会削弱闯关奖励感。设计接受在“选择、答对/答错、加载、通关”四个关键时刻强化反馈，但拒绝无意义循环动画。
- 前端挑战产品：每个状态都上复杂动画会增加包体和兼容风险。产品接受第一版只用 CSS 动画和 GPT 静态切图，不引入新动画运行库。
- 测试挑战前端：构建成功不能替代运行时证据。前端移除 Taro 内置 Button loading，保留自有水母加载层；复测后加载状态出现且无同类 DOM 异常。
- 设计挑战前端：手机端首屏标题挤压水母。前端缩小标题和插画占位，使标题收敛为两行，挑战卡完整露出。

### 负责人裁决

选择方案 B“克制游戏化”：

1. 水母主视觉保留在首页、加载、通关、报告四个关键节点。
2. 交互参考多邻国的即时反馈节奏，但不复制其视觉资产或重度动画。
3. 答对使用柔和黄色，答错使用弱红色；反馈固定在底部，讲解用可滚动半屏抽屉。
4. 连续动画仅保留水母轻浮动和气泡，按钮采用按压缩放；报告数据本身不循环跳动。
5. 图片压缩延后为独立性能任务，当前优先保留 GPT 插画画质；发布前必须解决包体告警。

结论：本地 Demo 迭代“有条件通过”，允许进入用户体验验收；不等同于公开发布批准。
